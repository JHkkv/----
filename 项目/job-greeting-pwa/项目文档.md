## 一、项目背景

### 用户画像
- 姓名：罗南杨
- 求职方向：内容运营/直播运营助理
- 城市：深圳（优先）→ 广州 → 泉州
- 经济状况：存款不足4000元，深圳生存窗口约1.3-1.6个月
- 核心痛点：每天手写10-20条BOSS招呼语，耗时1-2小时；不知道哪条招呼语效果好

### 原始需求
> "做一个简历软件，涵盖修改简历、生成打招呼语、自动搜索+一键投递，以Chrome插件形式用于BOSS直聘"

### 修正后定位
> **独立PWA + AI招呼语生成 + 简历分析**（不做Chrome扩展DOM操控、不做自动投递、不做自动搜索）

**原因**: 对抗审查发现 Chrome扩展TAM太小（BOSS用户80-95%移动端），全自动化=封号风险5/5，Boss官方AI功能随时可能覆盖。

### Phase 3: 安全加固（3 Agent）— "攻击模拟+修复验证"

| # | 审查维度 | 发现数 | 严重问题 |
|---|---------|--------|---------|
| 1 | API密钥+数据泄露 | 8项 | API Key localStorage明文；CDN无SRI；innerHTML防御纵深 |
| 2 | 输入验证+注入攻击 | 11项 | localStorage投毒可致崩溃；文件上传仅扩展名检查 |
| 3 | 隐私合规+网络传输 | 12项 | 隐私声明完全缺失；自定义API端点无HTTPS强制 |

#### 模拟攻击结果
| 攻击向量 | 修复前 | 修复后 |
|----------|--------|--------|
| `localStorage` 读API Key | ✅可窃取明文 | ❌ sessionStorage+btoa |
| `window.state.apiKey` | ✅全局暴露 | ❌ IIFE隔离 |
| XSS代码注入 | ❌ 已防御 | ❌ +CSP双重防线 |
| localStorage投毒致崩溃 | ✅ 成功白屏 | ❌ Array.isArray校验 |

#### 最终9项修复
1. API Key → sessionStorage + btoa编码
2. CDN脚本 → crossOrigin='anonymous'
3. localStorage加载 → Array.isArray类型校验
4. CSP → Content-Security-Policy meta标签
5. 隐私声明 → 页脚+弹窗
6. Rate Limit → 60秒最多3次
7. PII脱敏 → 手机号→138****
8. X-Frame-Options → DENY
9. IIFE隔离 → state不暴露到window

## 四、关键教训与陷阱

### 技术陷阱
1. **SSE手动解析不可靠** — 高速token流下丢字符/乱序，改用非流式 (`stream: false`)
2. **IIFE闭合要数括号** — 多一个`}`导致全部JS解析失败，页面完全无响应
3. **Service Worker不能做AI调用** — 30秒不活动即被回收，需用Offscreen Document
4. **TextDecoder需显式`utf-8`** — 默认自动编码可能导致中文乱码
5. **GitHub Pages需`.nojekyll`+workflow模式** — legacy branch模式会报错

### 战略教训
1. **先验证再开发** — 不要假设"更好的AI=更好的回复率"，需要真实用户数据
2. **做减法比做加法难** — 从"Chrome扩展+自动投递"缩减到"PWA+手动发送"是正确决策
3. **对抗审查不可跳过** — 13位专家都赞成的方案，3位审查者发现致命缺陷
4. **移动端优先** — BOSS用户80-95%在手机上，Web端Chrome扩展是错的方向
5. **不要向失业者收费** — 商业模式需重新思考

### 流程教训
1. **安全修复后立即验证** — IIFE修复引入的语法错误应立刻自测
2. **每次提交前做冒烟测试** — 打开页面点击每个按钮确保JS无语法错误
3. **SSH被墙时切HTTPS** — `git@github.com:22` 有时被墙，`https://`更稳定
