# Vibe Coding 项目上线前代码清理操作标准（AI执行规范）

## 一、核心原则

每次功能开发完成且测试通过后，AI必须主动执行墓碑代码清理。**先输出清单 → 人工确认 → 再删除 → 回归验证**。

## 二、触发时机

以下三个条件全部满足时，自动触发清理：

1. 目标功能全部开发完成
2. 对应正式测试用例全部执行通过
3. 人工实际验证功能运行正常、符合需求

> 开发调试阶段严禁提前删除临时代码，避免影响问题排查效率。

## 三、墓碑代码识别标准

以下5类属于墓碑代码，清理时必须纳入排查：

| 类别 | 特征 | 示例 |
|------|------|------|
| 一次性临时测试 | 为当场验证单个功能随手写的测试代码 | `practical-tests.js`、console粘贴脚本 |
| 调试打印语句 | 排查问题时临时加的打印，问题解决后无用 | 非关键链路的 `console.log` |
| 临时测试接口 | 跳过权限校验、无登录验证的后门接口 | 无鉴权的 debug 路由 |
| 写死的假数据 | 硬编码的假用户/假订单/假金额 | `const userId = "test-001"` |
| 一次性脚本 | 单次数据处理/环境配置用后即弃的脚本 | 硬编码路径的 OCR 脚本 |

### 红线（严禁误删）

- ✅ **正式回归测试代码**（需长期复用的测试用例）→ 保留
- ✅ **核心业务链路运行日志**（登录/支付/订单等关键节点）→ 保留
- ✅ **正式业务代码和配置** → 保留

> 拿不准的项单独标注，由人工决策。

## 四、标准操作流程

### 4.1 单轮功能开发后清理（日常迭代）

```
步骤1: AI输出清理清单 → 逐项说明文件路径、原始用途、当前必要性、建议操作
步骤2: 人工确认清单 → 存疑项由AI解释后人工拍板
步骤3: AI执行精准删除 → 不触碰红线保留项
步骤4: AI运行全部正式回归测试 → 无异常才算完成
```

### 4.2 全量历史清理（上线前兜底）

执行指令：
> 对当前项目做全量排查，检查所有遗留的临时测试代码、调试打印、无权限校验的临时测试接口、写死假数据、一次性脚本。
> 按文件维度输出清单，说明每项用途、风险等级、删除/保留建议。
> 重点标注无权限校验的临时接口和硬编码测试数据（高安全风险）。
> 清单输出后等待人工确认，再执行删除。

## 五、清理验收标准

满足以下全部条件，清理合格：

1. 所有已确认的墓碑代码均已清除，无高风险项遗漏
2. 项目全部正式回归测试用例执行通过
3. 正式回归测试代码、关键功能运行日志**完整保留**
4. 无未授权的临时测试接口、硬编码假数据残留

## 六、配套安全审查

清理过程中如发现以下安全问题，一并纳入清单报告：

- [ ] API Key/Token 存储方式（禁止明文base64存localStorage）
- [ ] CDN 外部脚本是否缺少 SRI 完整性哈希
- [ ] 是否有 `.next`/`dist` 等构建产物被提交到仓库
- [ ] 是否有数据库文件（.db/.sqlite 及 WAL/SHM）被提交
- [ ] 一次性脚本中是否有硬编码路径
- [ ] 文档中是否有指向已删除文件的过期引用
